# CRM Product + SOP Template Setup (Provisioning Prerequisites)

Before the auto-provisioning workflow (doc 12) can create an onboarding task for a LoRaWAN
node, the CRM needs three things: a **LoRaWAN product**, an **SOP template** for that product,
and a **hardware-catalog entry**. This guide shows how to create them via the CRM REST API and
explains why each is required.

All calls go to the live CRM backend `http://10.10.8.140:4000` with a bearer token from
`POST /auth/login`. `tools/provision-node.sh` performs the product + SOP-template steps
idempotently; this doc is the manual reference and the rationale.

## 0. Authenticate

```bash
BASE=http://10.10.8.140:4000
TOKEN=$(curl -s -X POST $BASE/auth/login -H 'Content-Type: application/json' \
  -d '{"email":"admin@southerneleven.com","password":"<password>"}' \
  | python3 -c 'import sys,json;print(json.load(sys.stdin)["access_token"])')
```

Seeded roles include `ADMIN` and `HARDWARE_ENGINEER` (both can provision).

## 1. Create a LoRaWAN product

```bash
curl -s -X POST $BASE/products -H "Authorization: Bearer $TOKEN" -H 'Content-Type: application/json' -d '{
  "name": "RAK4630 Environmental Node",
  "code": "LORA-ENV-RAK4630",
  "description": "RAK4630 + RAK14000 E-Ink + BME280, AS923 OTAA",
  "isLorawanProduct": true,
  "lorawanRegion": "AS923"
}'
```

`isLorawanProduct=true` is what makes the workflow register the device in ChirpStack at
`READY_FOR_INSTALLATION`; `lorawanRegion` documents the region. Capture the returned `id` as
`PRODUCT_ID`.

## 2. Create the SOP template (REQUIRED before task creation)

**Why:** `POST /workflow/tasks` returns `404 "SOP template for product <id> not found"` if the
product has no SOP template. The seeded products ship without one for new LoRaWAN products, so
you must add it.

```bash
curl -s -X POST $BASE/products/$PRODUCT_ID/sop-template \
  -H "Authorization: Bearer $TOKEN" -H 'Content-Type: application/json' -d '{
  "productId": "'$PRODUCT_ID'",
  "steps": [
    {"id":"step-1","title":"Provision & Join","description":"OTAA provisioning and join verification","order":1}
  ],
  "version": 1
}'
```

`SOPStepDto` fields: `id` (string), `title` (string), `description` (string), `order` (number),
optional `estimatedDuration` (minutes) and `requiredRole` (e.g. `HARDWARE_ENGINEER`). The path
`{id}` carries the product id; one minimal step is enough to satisfy task creation. Add more
steps to mirror a real install runbook.

## 3. Ensure a hardware-catalog entry

The `HARDWARE_PROCUREMENT_COMPLETE` and `HARDWARE_PREPARED_COMPLETE` transitions require a
`hardwareId`. List existing entries and reuse one, or create a RAK4630 entry:

```bash
curl -s $BASE/hardware-catalog -H "Authorization: Bearer $TOKEN" \
  | python3 -c 'import sys,json;[print(h["name"],h["id"]) for h in json.load(sys.stdin)]'
```

(The PoC reused `ESP32-WROOM-32` as a valid FK; the device's real identity is carried by its
serial + firmware + DevEUI. A dedicated `RAK4630` catalog entry under the `Microcontroller`
category is cleaner for production.)

## 4. The task lifecycle (where credentials enter)

Once 1–3 exist, `POST /workflow/tasks` (with `productId`) succeeds, and the task advances
through statuses. Two non-obvious points:

- **DevEUI/AppKey are supplied at `HARDWARE_PREPARED_COMPLETE`** in
  `deviceList[]` (`{hardwareId, deviceSerial, firmwareVersion, devEui, appKey}`). Nothing is
  auto-generated by the CRM — the caller (our script) generates and supplies them.
- **The pre-installation checklist has its own endpoint.** Passing
  `preInstallationChecklist` in the status-transition body is **ignored**; you must
  `PUT /workflow/tasks/{id}/pre-installation-checklist` with all 9 items `true`, otherwise
  `READY_FOR_INSTALLATION` rejects with "checklist incomplete (0/9)".

The full ordered REST sequence is implemented in `tools/provision-node.sh`.

## Status reference

Legal forward order:
`INITIALIZATION → SCHEDULED_VISIT → REQUIREMENTS_COMPLETE → HARDWARE_PROCUREMENT_COMPLETE →
HARDWARE_PREPARED_COMPLETE → READY_FOR_INSTALLATION → INSTALLATION_COMPLETE → VERIFICATION →
CLOSED`. **`READY_FOR_INSTALLATION`** triggers ChirpStack registration
(`WorkflowService.provisionLorawanDevices`).
